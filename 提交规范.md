# BATI 提交规范（简版）

## 分支与提交流程

1. 日常开发从 `main` 拉最新代码：`git pull`
2. 每个需求使用单独分支：`git checkout -b feat/xxx`
3. 完成后本地自检：
   - `npm run build`
   - `git status`
4. 提交后推送分支并发起合并

## Commit Message 规范

建议格式：`<type>: <简短说明>`

- `feat`: 新功能
- `fix`: 缺陷修复
- `style`: 样式调整（不改逻辑）
- `refactor`: 重构（不改行为）
- `docs`: 文档更新
- `chore`: 构建、脚本、配置调整

示例：

- `feat: 新增结果页核心风格标签`
- `fix: 修复球星图鉴图片显示不完整`
- `docs: 补充项目逻辑说明文档`

## 提交内容边界

- 一次提交尽量只做一类改动（功能 / 修复 / 文档分开）
- 不提交构建产物和缓存（`dist`、`*.tsbuildinfo`、`node_modules`）
- 提交前保证可运行，避免“半成品提交”

## 推送前检查清单

- `npm run build` 通过
- `git diff --staged` 确认无误提交
- 敏感信息未入库（`.env`、密钥、账号密码）
